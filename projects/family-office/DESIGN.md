# Family Office — Slack Multi-Agent Product Org Design

## 개요

이 문서는 기존 Family Office / C레벨 구조를 제품 개발 중심의 Slack multi-agent 조직으로 재편한 현재 설계를 정의한다.

핵심 레퍼런스는 세 가지다.
- Harness: 역할 분리, supervisor 구조, generate/validate 흐름
- Claude Managed Agents: 장기 작업, checkpoint, managed execution 철학
- IronAct Agents / Ops Lead: 운영 리드 중심의 역할 기반 협업 모델

이제 이 조직은 더 이상 CIO/CFO/CTO/COO 중심으로 설명하지 않는다.
기존에 만들어둔 봇 인스턴스는 재활용하되, 사용자-facing 역할과 채널 구조는 아래의 새 모델을 기준으로 운영한다.

## 목표

이 조직의 목표는 단순하다.
- 리가 Lead이자 PO로서 프론트도어를 맡는다.
- FE / BE / Designer / QA가 명확한 책임을 가진다.
- 실행 과정은 보이되 메인 채널을 시끄럽게 만들지 않는다.
- 중요한 결과물은 generate 후 validate를 거쳐 게시한다.
- 장기 작업, 병렬 작업, 이어서 하는 작업을 기본적으로 다룰 수 있게 만든다.

## 현재 운영 모델

### 역할 구조

현재 기준 역할은 아래 5개다.

1. 리 = Lead + PO
2. FE
3. BE
4. Designer
5. QA

즉 기존 5개 봇 슬롯을 유지하면서, 역할만 새 구조로 재할당한다.

### 기존 인스턴스 재활용 원칙

새 봇을 처음부터 다시 만들기보다, 기존에 만들어둔 봇 인스턴스와 프로필을 재활용한다.
다만 다음 원칙을 따른다.

- 내부 런타임 이름이나 레거시 프로필 이름은 일시적으로 남을 수 있다.
- 사용자-facing 이름, 역할, 운영 책임은 새 모델을 따른다.
- Slack 채널 구조와 대화 흐름은 새 역할 기준으로 재편한다.
- 안정화 전까지는 “레거시 슬롯 재사용 + 새 역할 부여” 방식으로 운영한다.

## 역할 정의

### 1) 리 — Lead + PO

리는 이 조직의 front door이자 운영 리드다.
동시에 Product Owner 역할도 맡는다.

핵심 책임:
- 사람 요청 intake
- 문제 정의와 scope 정리
- 우선순위 결정
- 누구에게 어떤 작업을 줄지 routing
- 병렬 작업 orchestration
- 산출물 synthesis
- 최종 사용자-facing 응답 정리
- acceptance 기준 초안 정의

리는 모든 구현을 직접 하는 역할이 아니다.
하지만 전체 문맥과 제품 방향, 우선순위를 가장 넓게 본다.

### 2) FE

FE는 사용자 인터페이스와 프론트엔드 구현을 담당한다.

핵심 책임:
- 화면 구현
- 컴포넌트 구조
- 상호작용 로직
- 상태 관리
- 사용자-facing polish
- Designer와 협업한 UX 반영
- QA 피드백 반영

### 3) BE

BE는 백엔드와 시스템 로직을 담당한다.

핵심 책임:
- API
- 데이터 모델
- 인증/권한
- 저장소/DB
- 워커/배치/서버 로직
- 외부 서비스 연동
- 성능/안정성 관련 구현

### 4) Designer

Designer는 UX, 화면 구조, 인터랙션 경험을 설계한다.

핵심 책임:
- 사용자 플로우 설계
- 화면 정보 구조
- UI 카피 초안
- 상호작용 톤 정리
- FE에 넘길 설계 의도 정리
- 필요 시 디자인 시스템 원칙 정리

### 5) QA

QA는 품질과 검증을 담당한다.

핵심 책임:
- acceptance criteria 기준 검증
- 테스트 시나리오 정리
- 회귀 체크
- edge case 확인
- release gate 역할
- “무엇이 끝났고 무엇이 아직 아닌지”를 분명히 말하기

## 협업 원칙

### 1. 리가 항상 시작점이다

기본적으로 사람의 요청은 리가 먼저 받는다.
사람이 매번 FE/BE/QA 중 누구를 불러야 하는지 고민하지 않게 한다.

### 2. 역할은 섞여 말하지 않는다

각 역할은 자기 역할에 맞는 형태로 말해야 한다.
예를 들면:
- 리: 방향, 우선순위, 결정, 요약
- FE: UI 구현 관점
- BE: API/로직/데이터 관점
- Designer: UX/플로우/카피 관점
- QA: 검증/리스크/완료 기준 관점

### 3. 최종 결론은 single-owner voice로 낸다

여러 역할이 병렬로 작업할 수는 있지만, 최종 사용자-facing 응답은 가능한 한 리가 정리한다.
필요하면 특정 역할이 최종 owner가 될 수 있지만, 동시에 여러 봇이 결론을 따로 말하는 방식은 피한다.

### 4. generate / validate / publish를 기본 흐름으로 쓴다

중요한 일은 기본적으로 아래 순서를 따른다.

1. Generate
- 초안 작성
- 조사
- 구현
- 분석

2. Validate
- 리뷰
- acceptance 확인
- 품질 체크
- 리스크 점검

3. Publish
- 리 또는 최종 owner가 결론을 정리해 전달

### 5. 실행은 보이되, 실행 로그는 분리한다

사용자는 과도한 내부 로그와 도구 중계를 싫어한다.
그래서 실행은 완전히 숨기지 않되, 메인 협업 채널과 실행 로그 채널을 분리한다.

원칙:
- 메인 채널: 결론, 방향, 요청, 상태 요약
- 실행 로그 채널: 긴 작업 로그, 중간 상태, 백그라운드 진행 흔적

## Slack 채널 구조

### 기본 채널

현재 목표 채널 구조는 아래와 같다.

- `#lead`
- `#product-room`
- `#fe`
- `#be`
- `#designer`
- `#qa`
- `#execution-log`
- `#shipping`

### 채널 의미

#### `#lead`
리의 메인 운영 채널.
우선순위 정리, 지시, 판단, 주요 상태 정리에 쓴다.

#### `#product-room`
역할 간 공용 협업 채널.
문제 정의, 방향 합의, 공유 브리핑, 큰 결정 정리에 쓴다.

#### `#fe`
프론트엔드 구현 논의와 작업 상태 공유.

#### `#be`
백엔드 구현 논의와 작업 상태 공유.

#### `#designer`
UX / 구조 / 카피 / 플로우 관련 논의.

#### `#qa`
검증, acceptance, 버그 triage, release gate 관련 논의.

#### `#execution-log`
긴 실행 로그, 백그라운드 작업, 툴 실행 흔적, 중간 진행 상황을 분리해 남기는 채널.

#### `#shipping`
배포, 릴리즈, 완료 공유, shipping note 기록 채널.

## 기존 채널에서 새 채널로의 전환

현재 Slack에 이미 존재하는 채널을 최대한 재활용한다.
기본 전환 방향은 아래와 같다.

- `#chief-of-staff`는 레거시 홈 채널로 유지한다.
- `#lead`를 신규 생성해 Lead + PO 전용 운영 채널로 사용한다.
- `#fo-lounge` → `#product-room`
- `#cio-general` → `#fe`
- `#cfo-general` → `#be`
- `#cto-general` → `#designer`
- `#coo-general` → `#qa`
- `#execution-log` → 신규 생성
- `#shipping` → 신규 생성

이 매핑은 기존 C레벨 의미를 유지하기 위한 것이 아니라, 기존 채널 슬롯을 새 역할 채널로 빠르게 전환하기 위한 실무적 매핑이다.

## 발화 정책

### 리
- 요청 해석
- 방향 제시
- 우선순위 결정
- 최종 요약
- 외부로 보일 결론 정리

### FE
- UI와 인터랙션 관점에서 말한다.
- 구현 상태는 짧고 명확하게 말한다.
- 디테일 로그는 필요 시 execution-log로 보낸다.

### BE
- API, 로직, 데이터 관점에서 말한다.
- 구조적 제약이나 리스크를 분명히 말한다.

### Designer
- UX와 정보 구조 관점에서 말한다.
- 불필요하게 미학적 잡담으로 흐르지 않는다.
- 구현 가능한 수준의 설계 언어를 사용한다.

### QA
- “됨/안됨”만 말하지 않는다.
- 무엇을 검증했고, 무엇이 남았고, 어떤 리스크가 있는지 말한다.

## 운영 정책

### 사용자 선호 반영

이 조직은 아래 사용자 선호를 기본 운영 원칙으로 반영한다.

- 스레드 강제를 최소화한다.
- 가능하면 채널 본문에서 자연스럽게 대화한다.
- 장황한 내부 툴 로그를 메인 응답에 노출하지 않는다.
- 짧고 자연스럽고 직접적인 응답을 선호한다.
- 복잡한 내부 조율은 내부적으로 하고, 사용자-facing 메시지는 요약해 전달한다.

### 스레드 정책

기본값은 스레드 최소화다.
다만 아래 경우에는 제한적으로 사용할 수 있다.
- 긴 검토가 한 메시지에 깊게 매달릴 때
- release note / bug triage처럼 특정 문맥을 묶는 편이 명확할 때
- 메인 채널 소음을 줄여야 할 때

## 실행 모델

### 1. 단기 작업
예:
- 작은 UI 수정
- API 한 개 추가
- 간단한 QA 확인

흐름:
- 리가 요청 수신
- 필요한 역할에 바로 라우팅
- 구현 / 검증
- 리가 최종 정리

### 2. 장기 작업
예:
- 기능 단위 개편
- 구조 변경
- 여러 단계 구현

흐름:
- 리가 목표와 완료 기준 정의
- FE / BE / Designer / QA로 작업 분해
- execution-log에 진행 상황과 checkpoint 기록
- 중간 산출물 기준으로 재개 가능하게 운영
- QA 또는 리가 최종 validate
- shipping에 완료 공유

### 3. 병렬 작업
예:
- Designer가 UX 정리하는 동안 FE는 컴포넌트 구조 준비
- BE는 API 계약 정리
- QA는 acceptance 시나리오 초안 작성

이 경우 리가 fan-out / fan-in을 담당한다.

## Validation / Release Gate

중요한 변경은 최소한 아래를 지나야 한다.

- 요구사항이 무엇인지 설명 가능해야 함
- 구현 범위가 무엇인지 설명 가능해야 함
- QA가 검증 기준을 말할 수 있어야 함
- 최종 owner가 “이제 보낼 수 있음”을 선언해야 함

release gate의 기본 owner는 QA지만,
최종 사용자-facing publish owner는 보통 리다.

## 장기적으로 추가할 것

이 문서는 현재 운영 전환의 첫 버전이다.
다음 문서/정책을 추가할 수 있다.

- role별 SOUL / operating prompt
- channel별 speaking policy
- shipping note template
- QA checklist
- product intake template
- execution-log checkpoint template

## 한 줄 요약

이 조직은 기존 Family Office C레벨 구조를 유지하려는 문서가 아니라,
기존 봇 인스턴스를 재활용해 리(Lead+PO), FE, BE, Designer, QA로 재편한 Slack-first multi-agent product org 설계 문서다.
